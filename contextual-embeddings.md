# Contextual Embeddings — What They Are & How to Add Them to This Stack

A practical guide to upgrading the platform's plain chunk embeddings to **contextual** ones — where the text you embed (and BM25-index) carries document-level context, so an isolated chunk like *"shall be punished with imprisonment for life"* still "knows" it belongs to **BNS s.103, punishment for murder**.

All numbers below are taken from the primary sources (Anthropic, Jina, the arXiv papers) and verified against them. All implementation patterns are wired to this repository's real code (`apps/api/app/{retrieval,embed,rerank,query}.py`, `apps/api/scripts/{ingest.py,init.sql}`).

---

## 0. TL;DR for this project

Your pipeline today embeds the **bare chunk**. The whole idea of contextual embeddings is to embed **context + chunk** instead. Your architecture is ideal for it because the change is almost entirely at **ingest time** — retrieval, RRF, rerank, and the `/api/{demo}/query` route barely change.

**Recommended rollout (cheapest → most impactful):**

1. **Contextual chunk headers** — prepend the `doc_title` + `section_ref` you *already store*. $0, no new dependency. Best bang on structured corpora (statutes, insurance, health).
2. **Anthropic Contextual Retrieval** — an LLM writes a per-chunk "situating" blurb; made cheap by **prompt caching**. Apply where headers under-deliver (support, French narrative).
3. **Contextual BM25 + reranking** — index the contextual text for lexical search too, and keep your cross-encoder rerank on top (you already have it).
4. **Late chunking** — only if you adopt a long-context embedding model; gives context with **no LLM call**.

Reported impact (Anthropic's own eval, "failure rate" = correct chunk missing from top-20):

| Configuration | Retrieval failure rate | Reduction |
|---|---|---|
| Baseline (dense + BM25) | 5.7% | — |
| + Contextual Embeddings | 3.7% | **−35%** |
| + Contextual Embeddings + Contextual BM25 | 2.9% | **−49%** |
| + Reranking | 1.9% | **−67%** |

## 1. What "contextual embeddings" means (5 distinct techniques)

The term is overloaded. People mean at least five different things:

### 1a. Anthropic "Contextual Retrieval" (Sept 2024)
A chunk pulled out of a document loses context. The fix: before embedding, prepend a short, **chunk-specific** blurb generated by an LLM that has seen the **whole document**. The same contextual text is also fed to the **BM25 index** ("Contextual BM25"). Then standard hybrid retrieval runs, optionally followed by reranking.

The blurb is "usually 50–100 tokens." Example transformation:

```
Original chunk:  "The company's revenue grew by 3% over the previous quarter."
Contextual chunk: "This chunk is from an SEC filing on ACME corp's performance in
                   Q2 2023; the previous quarter's revenue was $314 million.
                   The company's revenue grew by 3% over the previous quarter."
```

**The exact prompt Anthropic used** (with Claude 3 Haiku):

```
<document>
{{WHOLE_DOCUMENT}}
</document>
Here is the chunk we want to situate within the whole document
<chunk>
{{CHUNK_CONTENT}}
</chunk>
Please give a short succinct context to situate this chunk within the overall document for the purposes of improving search retrieval of the chunk. Answer only with the succinct context and nothing else.
```

Their headline config: contextual embeddings **+** contextual BM25 **+** reranking, retrieving **top-20** (better than top-5/10); for reranking they retrieved **top-150** then reranked down to **top-20**. ([Anthropic](https://www.anthropic.com/news/contextual-retrieval))

### 1b. Prompt caching makes it cheap
The naive cost is brutal: to contextualize every chunk you re-send the *entire parent document* once per chunk (50× the doc for a 50-chunk doc). **Prompt caching** fixes it — the document goes in a cached prefix written once and re-read at a steep discount per chunk. Anthropic's figure: **~$1.02 per million document tokens** (one-time, at ingest), under their assumptions (800-token chunks, 8k-token docs, 50-token instructions, 100-token context per chunk). Cache **reads** cost ~10% of base input (≈90% discount); **writes** ~25% more — so process **all chunks of a document back-to-back** to stay inside the cache TTL (~5 min). ([Anthropic prompt caching](https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching))

### 1c. Late Chunking (Jina AI, 2024) — context *without* an LLM
Late chunking **inverts the order of operations**:
1. Feed the **entire document** through a **long-context embedding model** to get **token-level** embeddings — every token's vector is conditioned on the whole document via self-attention.
2. **Then** apply chunk boundaries and **mean-pool the token embeddings within each chunk span** → one vector per chunk.

The result is a contextualized chunk vector **with no generation call** — it's a pooling trick on a single encoder pass. Jina demonstrate it with `jina-embeddings-v2` (8,192-token context) and `jina-embeddings-v3`. It produces similarity scores comparable to Anthropic's contextual embeddings **while avoiding the extra LLM**. Don't confuse it with **late *interaction*** (ColBERT), which stores many vectors per doc and scores at query time — late *chunking* keeps one vector per chunk (same storage as normal). ([Jina](https://jina.ai/news/late-chunking-in-long-context-embedding-models/), [arXiv:2409.04701](https://arxiv.org/abs/2409.04701))

### 1d. Contextual Document Embeddings (CDE) — Morris & Rush, 2024
A **training-time** idea, not an inference recipe. CDE (1) uses a **contextual contrastive objective** where the batch/negatives are corpus-aware (neighboring docs), and (2) a **contextual encoder** that conditions a document's embedding on neighboring documents — analogous to how IDF makes term weights corpus-relative, but learned. Reports **SOTA on MTEB for its size class**, with the gap largest **out-of-domain**. *Relevance here:* informational — you'd benefit only by **adopting a model trained this way**, not by toggling something over `bge-m3`. ([arXiv:2410.02525](https://arxiv.org/abs/2410.02525), ICLR 2025)

### 1e. Contextual chunk headers (the cheap heuristic — start here)
Instead of paying an LLM, **prepend deterministic metadata you already have** — the document title + section path (e.g. `"Bharatiya Nyaya Sanhita 2023 — BNS s.103:"`). On **structured** corpora this captures much of the Anthropic gain for **$0 and zero added latency**, because the structure already disambiguates the chunk. Your `chunks` table already stores `doc_title` and `section_ref`, and `manifest.yaml` carries `doc_shorts`/section formats — so you have the raw materials with no new extraction. **This is the right Phase 1.**

## 2. How to implement each (wired to your code)

### 2.0 Schema: store contextual text, keep the original for display
Add a column for the embedded/indexed text; keep `content` as the human-readable original (the API returns `content[:400]` to the UI and builds citations from it — display must stay clean).

```sql
-- migration
ALTER TABLE chunks ADD COLUMN IF NOT EXISTS content_ctx TEXT;
UPDATE chunks SET content_ctx = content WHERE content_ctx IS NULL;  -- backfill
ALTER TABLE chunks ALTER COLUMN content_ctx SET NOT NULL;
```

Rebuild the BM25 index over `content_ctx` (this is what makes it *Contextual* BM25). In `init.sql`:

```sql
-- was: USING bm25 (id, content, doc_title, section_ref, demo_id, visibility)
CREATE INDEX IF NOT EXISTS chunks_bm25 ON chunks
    USING bm25 (id, content_ctx, doc_title, section_ref, demo_id, visibility)
    WITH (key_field='id');
```

The dense `embedding` column is unchanged in shape — only the **string fed to `embed_doc()`** changes. In `app/retrieval.py:_bm25()`, match against the contextual column but still `SELECT content` for display:

```python
"WHERE content_ctx @@@ %s AND demo_id = %s AND visibility = ANY(%s)"
```

Re-run `scripts/check_pg_search_fusion.sh` afterward to confirm `demo_id`/`visibility` still fuse into the index walk.

### 2.1 Phase 1 — contextual headers (zero LLM), driven by your manifest
You already compute `section_ref` and have `doc_title`. In `ingest_doc()`, between `sub_chunk()` and `embed_doc()`:

```python
def chunk_header(doc_title: str, section_ref: str) -> str:
    return f"[{doc_title} — {section_ref}]"

ctx_texts = [
    f"{chunk_header(doc_title, c['section_ref'])}\n\n{c['text']}"
    for c in all_chunks
]
vecs = embed_doc(ctx_texts)            # embed header + chunk
# store: content = c["text"] (clean),  content_ctx = ctx_texts[i] (embedded/indexed)
```

This mirrors the format `query.py:_build_context()` already uses (`[{section_ref} — {doc_title}]`), keeping the system consistent end-to-end. For statutes, enrich the header with chapter/part when the manifest exposes it.

### 2.2 Phase 2 — Anthropic per-chunk context (Haiku + prompt caching)
This slots into the **same insertion point**. Use the Anthropic SDK directly: your runtime `gateway.py` calls OpenRouter, which does **not** expose Anthropic `cache_control`, so the cost-saving cache only works against the Anthropic API. Contextualization is an **offline ingest job**, so a second SDK is fine.

```python
# scripts/contextualize.py  (ingest-time only)
import anthropic
client = anthropic.Anthropic()  # ANTHROPIC_API_KEY

SITUATE = (
    "Here is the chunk we want to situate within the whole document\n"
    "<chunk>\n{chunk}\n</chunk>\n\n"
    "Please give a short succinct context to situate this chunk within the "
    "overall document for the purposes of improving search retrieval of the "
    "chunk. Answer only with the succinct context and nothing else."
)

def contextualize(full_doc: str, chunks: list[str],
                  model: str = "claude-haiku-4-5") -> list[str]:
    out = []
    for chunk in chunks:
        msg = client.messages.create(
            model=model, max_tokens=120, temperature=0.0,
            messages=[{"role": "user", "content": [
                {   # CACHED: whole document, written once, re-read per chunk
                    "type": "text",
                    "text": f"<document>\n{full_doc}\n</document>",
                    "cache_control": {"type": "ephemeral"},
                },
                {"type": "text", "text": SITUATE.format(chunk=chunk)},  # not cached
            ]}],
        )
        out.append(msg.content[0].text.strip())
    return out
```

Wire into `ingest_doc()`:

```python
blurbs = contextualize(raw, [c["text"] for c in all_chunks])  # one cached doc, N cheap reads
ctx_texts = [f"{b}\n\n{c['text']}" for b, c in zip(blurbs, all_chunks)]
vecs = embed_doc(ctx_texts)                                   # embed blurb + chunk
```

Correctness notes:
- **Process a document's chunks back-to-back** so the cached prefix stays warm (TTL ~5 min).
- **Embed `context + chunk`, not the blurb alone** — the blurb disambiguates, the chunk carries the answer.
- Keep `temperature=0`, `max_tokens ≈ 80–120` → short, deterministic, cheap blurbs.
- **Re-ingest is required** (embeddings change). Cache blurbs to disk keyed by `doc_id + chunk_index + hash(chunk)` so re-runs don't re-bill the LLM.
- Drive it per demo from `manifest.yaml` (`contextualization: headers | llm | none`) — statutes use `headers`, support/French use `llm`.

### 2.3 Contextual BM25 + RRF (mostly unchanged)
Both legs now operate on contextual text: the dense vector comes from `content_ctx` embeddings; BM25 matches the `content_ctx` index (§2.0). **`_rrf()` stays exactly as is** (`RRF_K = 60`).

RRF in one line: an item's fused score is `Σ 1/(K + rank)` across the lists it appears in. `K=60` damps top-rank dominance and needs **no score normalization** across heterogeneous scorers (cosine vs BM25) — which is exactly why it suits dense+BM25 fusion. Your `scores[rid] += 1.0 / (RRF_K + rank + 1)` is textbook-correct ([Cormack et al., 2009](https://plg.uwaterloo.ca/~gvcormack/cormacksigir09-rrf.pdf)). Optional enhancement: **weighted RRF** (`w_dense`, `w_bm25`) tuned on your golden set.

### 2.4 Reranking — why it stacks
A reranker is a **cross-encoder**: it reads `(query, chunk)` together and scores relevance directly — far more accurate than bi-encoder cosine, but too slow over the whole corpus. So: cheap recall (hybrid) fetches `top_k=20`; the reranker keeps `top_n=5`. **This is exactly what `app/rerank.py` + `query.py` already do.** It stacks with contextual retrieval because they fix different failures:
- Contextual embeddings/BM25 fix **recall** — getting the right chunk into the candidate set.
- Reranking fixes **precision/order** — putting the best candidate on top.

That's why Anthropic's **−67%** appears only when rerank is added on top of contextual hybrid. For your multilingual (French + Indian-language) docs, evaluate `BAAI/bge-reranker-v2-m3` or Cohere `rerank-v3.5` against your current NVIDIA reranker. A/B feeding the reranker `content` vs `content_ctx`.

### 2.5 Late chunking (alternative — no LLM, but a model swap)
Requires a long-context embedding model exposing token-level outputs:

```python
tok = tokenizer(full_doc, return_offsets_mapping=True, truncation=True, max_length=8192)
token_embs = model(**tok).last_hidden_state[0]              # [num_tokens, dim], context-aware
chunk_vecs = [token_embs[s:e].mean(0) for (s, e) in chunk_spans]   # mean-pool each span
chunk_vecs = [v / v.norm() for v in chunk_vecs]            # normalize (pipeline expects it)
```

Practicalities: map your 512-token boundaries onto the long-context tokenizer offsets; for docs longer than the window, process overlapping macro-windows (e.g. 8k with 512 overlap); keep `EMBED_DIM=1024` to match your `vector(1024)` column. **No LLM call**, but it **swaps the embedding model** → re-ingest + re-validate the bge-m3 query-prefix asymmetry (`scripts/check_embed_asymmetry.py`). Treat as a separate project gated on a clear win.

## 3. Cost, latency & complexity

| Technique | LLM call? | Where cost lands | Marginal cost | Query latency | Re-ingest? |
|---|---|---|---|---|---|
| Contextual headers (1e) | No | none | **$0** | none | yes |
| Anthropic contextual (1a/1b) | Yes (Haiku + caching) | one-time, ingest | **~$1.02 / M doc-tokens** | **none** | yes |
| Late chunking (1c) | No | one-time, ingest (encoder pass) | GPU/inference | none | yes (+ model swap) |
| CDE (1d) | No (train-time) | train / adopt checkpoint | n/a | none | yes (+ model swap) |
| Reranking (stacks on all) | Yes (query-time) | per query | rerank model | **+rerank_ms** | no (already wired) |

Key points:
- **All contextualization is index-time; none adds query latency.** The only query-time cost is reranking — already in your `latency.rerank_ms`.
- **Prompt caching is what makes 1a affordable.** Without it you'd re-send the doc per chunk (~10× more).
- **Anthropic also notes:** if a knowledge base is **< ~200k tokens (~500 pages)**, skip retrieval entirely and just put the whole corpus in the prompt with caching. Each of your *individual* statutes is small, but the law corpus as a whole (and the multi-demo total) exceeds this, so retrieval is the right architecture — but it's a useful sanity check per demo.

## 6. Why this fits your architecture cleanly

- **One change-point.** Contextualization lives in `scripts/ingest.py` between `sub_chunk()` and `embed_doc()`. `retrieval.py` (`_rrf`), `rerank.py`, and `/api/{demo}/query` are unchanged.
- **Display stays clean.** Keep the original in `content` (UI truncates `content[:400]`; citations come from `section_ref`/`doc_title`); embed/index `content_ctx`. No citation regressions.
- **Contextual BM25 is one index edit** + the `@@@` predicate swap; re-run `check_pg_search_fusion.sh`.
- **Per-demo control** via `manifest.yaml`, matching the existing per-demo hooks pattern.
- **Asymmetry is safe.** `embed.py` notes bge-m3 asymmetry comes only from the query prefix. Contextualization touches the **passage** side only, so the mechanism is unaffected — but re-validate it if you adopt late chunking's new model.

## 7. Recommended phased rollout

1. **Phase 1 — Contextual headers.** $0, ingest-only. Prepend `[{doc_title} — {section_ref}]` (+ chapter where available) to `content_ctx`; embed + Contextual-BM25 on it. **Gate:** recall@20 / nDCG@5 vs baseline.
2. **Phase 2 — Anthropic LLM context + caching.** Add `scripts/contextualize.py` (Haiku, cached doc prefix, ~80–120-token blurbs), cache blurbs to disk. Apply where headers underperform. **Gate:** does it beat headers enough to justify ~$1/M-doc-tokens + re-ingest?
3. **Phase 3 — Confirm rerank stacks** on the contextual pipeline (it already runs after retrieval); A/B `content` vs `content_ctx`; evaluate `bge-reranker-v2-m3` for multilingual. **Gate:** marginal nDCG@5 lift.
4. **Phase 4 — Late chunking** only if cost/long-docs push you there; pilot `jina-embeddings-v3` on one demo as a separate, eval-gated project.

**Net:** Phases 1–3 reuse all your existing retrieval/RRF/rerank machinery and only change *what text you embed and BM25-index at ingest time* — the lowest-risk path to the recall gains contextual retrieval is known for.
